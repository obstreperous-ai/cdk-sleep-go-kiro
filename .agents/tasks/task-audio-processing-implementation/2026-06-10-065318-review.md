# Full audio processing pipeline for the Lambda processor

The Lambda processor has been upgraded from a validation-only stub to a complete audio processing pipeline: download from S3, synthesize via Polly, upload to the output bucket, and update DynamoDB with the result. The CDK layer grants the new S3 read permission and passes `INPUT_BUCKET_NAME` to the Lambda environment. Interface-based DI keeps all three AWS clients mockable, and 13 unit tests exercise both the happy path and each failure mode individually.

Watch for: The downloaded S3 object is never used — Polly generates fresh audio regardless of the input file content (confirmed). The `INPUT_BUCKET_NAME` env var is set but never read by the Lambda code (confirmed). DynamoDB update failure on the COMPLETED path surfaces as a Lambda error, contradicting the ARCHITECTURE.md claim that it's "logged but processing is considered complete" (confirmed).

## High-level view

The processing pipeline reads from S3 only to prove connectivity, then discards the input entirely. Polly always generates the same hardcoded phrase regardless of what was uploaded. This makes the "download" step a no-op in terms of data flow — the audio content flowing through the pipeline is purely synthetic. This may be intentional as a scaffold, but the architecture docs describe it as a real processing pipeline.

Infrastructure changes are minimal and correct: one `GrantRead` call and one additional env var. The CDK tests mirror the existing output-bucket permission tests exactly.

Error handling follows a consistent pattern — every failure marks the DynamoDB record as FAILED before returning an error — except the final DynamoDB update itself, which returns an error to the caller rather than swallowing it as the docs promise.

The output key uses a timestamp formatted as `20060102T150405Z` rather than the Unix-seconds format described in ARCHITECTURE.md (`processed/{audioId}/{timestamp}.mp3` where "timestamp is a Unix timestamp in seconds").

1. **Downloaded S3 object is discarded** — The `GetObject` result body is deferred-closed but never consumed. The pipeline produces Polly output regardless of the input file. If this is intentional scaffolding, the architecture docs should say so rather than describing a "download → process" flow.
2. **INPUT_BUCKET_NAME env var unused** — The Lambda receives `INPUT_BUCKET_NAME` in its environment but never reads it; it trusts whatever bucket name arrives in the event payload. Either validate the event bucket against the env var or remove the env var to avoid confusion.
3. **DynamoDB failure on COMPLETED path returns error** — `updateDynamoDBStatus` failure after successful S3 upload propagates as a Lambda error, triggering the Step Functions Catch and `MarkFailed`. 4. **Output key timestamp format diverges from docs** — Code uses `20060102T150405Z` (ISO-like); ARCHITECTURE.md says "Unix timestamp in seconds." One of them should change to match the other.
5. **Polly audio read into memory without size guard** — `io.ReadAll(pollyOutput.AudioStream)` has no upper-bound check. Polly responses are bounded in practice (~5 min max), but a defensive `io.LimitReader` would prevent memory exhaustion if the stream misbehaves.

## The "download then discard" data flow

The processor calls `S3 GetObject` on the input bucket and immediately defers closing the body. The returned `io.ReadCloser` is never read. The next step unconditionally calls Polly with a hardcoded text string:

```go
getOutput, err := p.S3Client.GetObject(ctx, &s3.GetObjectInput{
    Bucket: aws.String(event.Bucket),
    Key:    aws.String(event.ObjectKey),
})
if err != nil { ... }
if getOutput.Body != nil {
    defer getOutput.Body.Close()
}

// Step 2: Call Polly — no reference to getOutput content
pollyOutput, err := p.PollyClient.SynthesizeSpeech(ctx, &polly.SynthesizeSpeechInput{
    Text: aws.String("Welcome to your sleep audio session. Relax and breathe deeply."),
    ...
})
```

The uploaded artifact is always Polly's output, never a transformation of the input file. The S3 download effectively becomes a "file exists" check that also acquires IAM costs and latency. If the intent is scaffolding for a future step that merges input audio with synthesized narration, the code comment should say so — currently it reads as though the download feeds the pipeline.

## INPUT_BUCKET_NAME: granted but never consumed

The CDK passes `INPUT_BUCKET_NAME` into the Lambda environment and grants `s3:GetObject` on the input bucket. The Lambda's `main()` reads `TABLE_NAME` and `OUTPUT_BUCKET_NAME` but never reads `INPUT_BUCKET_NAME`:

```go
tableName := os.Getenv("TABLE_NAME")
outputBucket := os.Getenv("OUTPUT_BUCKET_NAME")
```

The `Process` method trusts `event.Bucket` verbatim. An attacker who can inject events (or a misconfigured EventBridge rule) could point the Lambda at any bucket the role has access to. Reading and validating `INPUT_BUCKET_NAME` against the event would close this gap.

## DynamoDB update failure asymmetry

On the error paths (S3 download fail, Polly fail, S3 upload fail), the code calls `updateDynamoDBStatus(ctx, ..., "FAILED", ...)` and then returns an error — meaning DynamoDB failure inside that helper is silently ignored (the helper's error return value is discarded on those paths). On the *success* path, the DynamoDB update failure propagates as the Lambda's return error:

```go
err = p.updateDynamoDBStatus(ctx, event.AudioID, "COMPLETED", outputLocation, fileSize)
if err != nil {
    return Response{}, fmt.Errorf("failed to update DynamoDB: %w", err)
}
```

This creates two problems. First, a transient DynamoDB throttle after successful processing causes the Step Functions Catch clause to fire, which may re-mark the record as FAILED — overwriting the audio that was already correctly uploaded. Second, this contradicts ARCHITECTURE.md's statement: "If DynamoDB update fails after successful processing, the error is logged but processing is considered complete." The code does not log-and-continue; it returns an error.

## Memory-unbounded Polly stream read

```go
audioData, err = io.ReadAll(pollyOutput.AudioStream)
```

Polly limits synthesis to 3,000 characters of input text (or 6,000 with SSML), so in practice the stream will never be large. But `io.ReadAll` on an `io.Reader` returned by a network call, with no `LimitReader` wrapper, is a pattern that can bite if the interface is ever reused for a longer synthesis or if Polly's response headers are malformed. This is a low-severity concern but worth noting for defensive coding.

## Output key timestamp format

Code generates the key as:

```go
timestamp := time.Now().UTC().Format("20060102T150405Z")
outputKey := fmt.Sprintf("processed/%s/%s.mp3", event.AudioID, timestamp)
```

This produces keys like `processed/abc123/20260610T065318Z.mp3`. ARCHITECTURE.md states the convention is `processed/{audioId}/{timestamp}.mp3` where "timestamp is a Unix timestamp in seconds at the time of processing," implying something like `processed/abc123/1718000000.mp3`. The difference is cosmetic but creates a doc/code mismatch that will confuse anyone writing queries or lifecycle rules against the output prefix.

## Test coverage

The 13 tests cover: input validation (5 valid extensions, 3 empty fields, 4 invalid extensions), structured logging, successful end-to-end processing, S3 download failure, Polly failure, S3 upload failure, DynamoDB update failure, response field assertions, Polly parameter verification, S3 GetObject parameter verification, and DynamoDB update expression verification.

Not tested: the `main()` function's config loading (expected — integration concern), behavior when `pollyOutput.AudioStream` is nil (the code doesn't guard against this), and the scenario where `updateDynamoDBStatus` fails on the error-path calls (it's silently ignored, which may or may not be desired).

## CDK infrastructure changes

The CDK diff is two lines: `inputBucket.GrantRead(processorLambda, nil)` and adding `"INPUT_BUCKET_NAME"` to the environment map. Both CDK tests follow the established pattern exactly. The `GrantRead` produces `s3:GetObject`, `s3:GetBucketLocation`, and `s3:ListBucket` — `ListBucket` isn't needed for single-object downloads but is CDK's default bundle and not worth fighting.
