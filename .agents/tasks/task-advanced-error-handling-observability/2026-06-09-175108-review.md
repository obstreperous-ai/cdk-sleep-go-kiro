# Retry policies, X-Ray tracing, and CloudWatch alarms for the audio pipeline

The change adds retry-with-backoff on ProcessAudio and PollyTask, specific error-type Catch blocks before the States.ALL fallback, Lambda X-Ray active tracing, two CloudWatch alarms (ExecutionsFailed and Lambda Errors), and structured JSON logging in the Lambda handler. Documentation and a Mermaid diagram update round it out.

Watch for: ProcessAudio carries **two overlapping retry blocks** with different configs for the same error types (confirmed). CloudWatch alarms use `Average` statistic on count-based metrics—should be `Sum` (confirmed). Alarms have no `AlarmActions`, so they fire but never notify anyone (confirmed).

## High-level view

The state machine retry strategy layers a CDK-generated default retry (6 attempts on Lambda service errors) with a second manually-added retry on partially-overlapping error types. Step Functions evaluates retry entries in array order, so the first matching entry wins — the second block's `Lambda.ServiceException`, `Lambda.AWSLambdaException`, and `Lambda.SdkClientException` entries are unreachable. Only `States.TaskFailed` in the second block actually adds coverage.

The CloudWatch alarms are structurally correct (right namespaces, dimensions, thresholds) but functionally inert: no `AlarmActions` are configured, so the alarms transition to ALARM state without sending notifications. The ARCHITECTURE.md claims they route to the SNS Failed Topic, but the synthesized template has no such wiring.

Both alarms use `Statistic: Average` for metrics that emit integer counts (0 or 1 per data point). `Sum` is the standard idiom for count metrics and makes the intent unambiguous. Using Average could produce misleading values under concurrent invocations—1 error in 100 invocations gives Average = 0.01, below the threshold of 1.

## Duplicate retry entries on ProcessAudio

The CDK `LambdaInvoke` construct automatically adds a retry entry for `[Lambda.ClientExecutionTimeoutException, Lambda.ServiceException, Lambda.AWSLambdaException, Lambda.SdkClientException]` with `IntervalSeconds: 2`, `MaxAttempts: 6`, `BackoffRate: 2`. The code then calls `processAudio.AddRetry` with `[Lambda.ServiceException, Lambda.AWSLambdaException, Lambda.SdkClientException, States.TaskFailed]` at `IntervalSeconds: 3`, `MaxAttempts: 3`, `BackoffRate: 2`.

Step Functions evaluates retry entries in array order. The first entry that matches an error wins. Because the CDK-default entry appears first and already matches three of the four error types in the second block, the second entry only ever fires for `States.TaskFailed`. The net effect:

- Lambda service errors retry 6 times at 2s base (CDK default), not 3 times at 3s (manual intent).
- `States.TaskFailed` retries 3 times at 3s (manual entry — this one works as intended).

The synthesized definition confirms this:

```json
"Retry":[
  {"ErrorEquals":["Lambda.ClientExecutionTimeoutException","Lambda.ServiceException","Lambda.AWSLambdaException","Lambda.SdkClientException"],"IntervalSeconds":2,"MaxAttempts":6,"BackoffRate":2},
  {"ErrorEquals":["Lambda.ServiceException","Lambda.AWSLambdaException","Lambda.SdkClientException","States.TaskFailed"],"IntervalSeconds":3,"MaxAttempts":3,"BackoffRate":2}
]
```

To get the stated 3-attempt/3s behavior, either suppress the CDK default retry (pass `RetryOnServiceExceptions: jsii.Bool(false)` in `LambdaInvokeProps`) and replace it with a single comprehensive retry, or trim the manual retry to only `States.TaskFailed`.

## CloudWatch alarms are notification-inert

The CDK code creates both alarms with `awscloudwatch.NewAlarm` but never calls `AddAlarmAction`. The `failedTopic` SNS topic exists in the same stack and is available at the point where alarms are created. The fix:

```go
snsAction := awscloudwatchactions.NewSnsAction(failedTopic)
stateMachineAlarm.AddAlarmAction(snsAction)
lambdaAlarm.AddAlarmAction(snsAction)
```

This requires importing `awscloudwatchactions`. Without this, the alarms are visible in the CloudWatch console but produce no operational alerts.

## Alarm statistic mismatch

Both alarms use CDK's default `Average` statistic. For `ExecutionsFailed` and `Errors` (count metrics), `Sum` is correct. With a 1-minute period and a single invocation, Average equals Sum. But under load—1 error in 100 invocations within the period—`Average` yields 0.01, well below the threshold of 1. The alarm would never fire despite real errors occurring.

## Test coverage gaps

Seven new CDK tests verify retries, X-Ray, alarms, and specific error types. Not tested:

- The alarm `Statistic` value (tests assert `MetricName` and `Namespace` but not `Statistic`).
- Alarm actions (no assertion that alarms route to SNS).
- The retry `IntervalSeconds`, `MaxAttempts`, or `BackoffRate` values (tests only assert the `Retry` keyword is present adjacent to the task name, not the configuration values).
- The duplicate retry block behavior — a test that asserts a specific retry count or interval would surface this.
