# SNS notification tasks for pipeline completion and failure

This change adds two encrypted SNS topics and wires SNS Publish tasks into the Step Functions state machine as terminal states on both the success and failure paths. The success chain becomes WriteInitialRecord → PollyTask → MarkCompleted → NotifyCompleted; the failure chain becomes MarkFailed → NotifyFailed. Both topics are encrypted with the AWS-managed SNS KMS key. Six CDK assertion tests validate the resource count, encryption, IAM permissions, and state machine definition content.

**Watch for:** The notification tasks have no retry or catch configuration, so a transient SNS failure causes the entire state machine execution to fail — even though the DynamoDB record already reflects the correct status (confirmed). Hardcoded topic names prevent multi-stack deployment without conflict (confirmed).

## High-level view

Notifications are placed after the DynamoDB status updates, so the metadata table remains the source of truth regardless of notification delivery. The failure notification includes error context from the Catch clause's `$.error` result path.

The DynamoDB tasks have retry policies; the SNS tasks don't. Since these are terminal states with no Catch, an SNS outage or throttle leaves the execution in a FAILED state even though the pipeline work completed successfully. This asymmetry is the primary concern.

The test suite confirms resource existence and IAM policy shape but doesn't assert state machine transition ordering.

## Notification task resilience gap

The DynamoDB tasks (`MarkCompleted`, `MarkFailed`) are configured with retry policies: all errors, 3 attempts, 2-second interval, exponential backoff. The SNS Publish tasks have no retry and no catch.

```go
// DynamoDB tasks get retry:
markCompleted.AddRetry(&awsstepfunctions.RetryProps{
    Errors:      &[]*string{awsstepfunctions.Errors_ALL()},
    Interval:    awscdk.Duration_Seconds(jsii.Number(2)),
    MaxAttempts: jsii.Number(3),
    BackoffRate: jsii.Number(2.0),
})

// SNS tasks get nothing
```

Since NotifyCompleted is the terminal state of the success chain, a transient SNS failure (throttle, KMS key unavailable) causes the execution to end in FAILED status in Step Functions — while the DynamoDB record correctly shows COMPLETED. Operators monitoring execution failures will investigate ghost failures.

The fix: add the same `AddRetry` block to both SNS tasks. Additionally, add a Catch that transitions to a terminal Succeed state with a result indicating notification delivery failed, so the execution still completes successfully.

## Failure path error context

The `NotifyFailed` message references `$.error`:

```go
"error": awsstepfunctions.JsonPath_StringAt(jsii.String("$.error")),
```

The Catch clause sets `ResultPath: jsii.String("$.error")`, placing the entire Step Functions error object (with `Error` and `Cause` fields) at that path. `JsonPath_StringAt` will serialize this as a JSON string in the SNS message, so subscribers receive a stringified JSON object rather than structured fields. Any downstream consumer parsing error details will need to handle this double-encoding.

## Hardcoded physical topic names

Both topics specify explicit `TopicName` values:

```go
TopicName: jsii.String("SleepAudioPipelineCompleted"),
TopicName: jsii.String("SleepAudioPipelineFailed"),
```

Physical names in CDK prevent deploying multiple instances of the stack in the same account/region (staging + production, or PR-specific test stacks). They also block seamless resource replacement during updates — CDK can't create a new topic before deleting the old one if the name is fixed. Omitting `TopicName` (letting CDK generate unique names) or adding a stack-level prefix resolves both issues.

## Test coverage gaps

Not tested: the transition graph itself. The tests assert that "NotifyCompleted" and "NotifyFailed" appear somewhere in the definition, but don't verify that `MarkCompleted` transitions to `NotifyCompleted` or that `MarkFailed` transitions to `NotifyFailed`. A refactor that accidentally wires NotifyCompleted after WriteInitialRecord would still pass all tests. Testing the `Next` field of specific states in the definition string would catch ordering regressions.
