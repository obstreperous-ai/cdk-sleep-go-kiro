# DynamoDB metadata tracking for the sleep audio pipeline

This change adds a DynamoDB table (`SleepAudioMetadataTable`) to record the lifecycle of each audio processing job and wires it into the Step Functions state machine as PutItem/UpdateItem tasks bracketing the existing Polly call. The state machine now writes a `PROCESSING` record before Polly runs, updates it to `COMPLETED` on success, and catches Polly failures to mark the record `FAILED`. IAM permissions are scoped to the table ARN for DynamoDB actions, and five new CDK assertion tests validate the resource properties, definition content, and permissions.

**Watch for:** `updatedAt` uses the EventBridge event timestamp (`$.time`) rather than the current wall-clock time, so it reflects *when the upload happened*, not *when processing finished* (confirmed). The MarkCompleted and MarkFailed states themselves have no error handling — if DynamoDB is unreachable at that point, the execution fails with metadata stuck in `PROCESSING` (confirmed).

## High-level view

The DynamoDB table is on-demand with a string partition key, PITR, and AWS-managed encryption. No sort key or GSIs — sufficient for keyed lookup by `audioId` but not for querying by status or time range, which future milestones will likely need.

The state machine chain (WriteInitialRecord → Polly → MarkCompleted, with Catch → MarkFailed) preserves the original EventBridge payload via `ResultPath` across all transitions. The DynamoDB update states are unprotected terminal tasks — a transient DynamoDB error during completion marking leaves metadata inconsistent with no automated recovery.

The `updatedAt` value in both MarkCompleted and MarkFailed is sourced from `$.time`, the EventBridge event timestamp baked into the execution input. This is a static value that won't reflect actual completion/failure time, especially if Polly runs for multiple seconds.

Tests don't verify ordering of states in the chain or the specific Catch transition target — a regression in state machine topology (e.g., accidentally swapping MarkCompleted and MarkFailed) would not be caught.

## Timestamp semantics in update states

Both MarkCompleted and MarkFailed derive `updatedAt` from `$.time`:

```go
":updatedAt": awsstepfunctionstasks.DynamoAttributeValue_FromString(
    awsstepfunctions.JsonPath_StringAt(jsii.String("$.time"))),
```

In the synthesized ASL this becomes `"S.$": "$.time"`. The `$.time` field is the ISO-8601 timestamp from the EventBridge event payload, set at event emission time. For `createdAt` in WriteInitialRecord this is semantically appropriate — it's when the upload was detected. For `updatedAt` in MarkCompleted, it represents the same moment, not when Polly finished.

Step Functions exposes `$$.State.EnteredTime` (the context object) which gives the wall-clock time when the current state began execution. Using this for `updatedAt` would accurately reflect when the status transition occurred.

## Error handling gap at terminal states

MarkFailed and MarkCompleted are unguarded DynamoDB calls. The synthesized ASL shows no `Retry` block on either:

```json
"MarkCompleted":{"End":true,"Type":"Task","ResultPath":null,"Resource":"arn:...dynamodb:updateItem",...}
"MarkFailed":{"End":true,"Type":"Task","ResultPath":null,"Resource":"arn:...dynamodb:updateItem",...}
```

A transient DynamoDB error (`InternalServerError`, network timeout) on either state causes the execution to fail with the metadata record stuck in `PROCESSING` indefinitely. Adding a `Retry` with `IntervalSeconds: 2, MaxAttempts: 3, BackoffRate: 2` for `DynamoDB.InternalServerError` and `DynamoDB.ServiceUnavailableException` would make the completion path resilient.

## Partition key design and overwrite semantics

The table uses `audioId` (the S3 object key) as the sole partition key. WriteInitialRecord performs an unconditional PutItem — no `ConditionExpression` preventing overwrites. Since the input bucket has versioning enabled, the same key can be uploaded multiple times. Each upload triggers a new execution, and the second execution's PutItem silently overwrites the first record (potentially mid-flight if the first is still processing).

Two concurrent executions for the same key can corrupt each other's status tracking: execution A writes PROCESSING, execution B writes PROCESSING (overwriting A's record), execution A writes COMPLETED (overwriting B's PROCESSING status). A conditional write (`attribute_not_exists(audioId) OR #s = :completed`) or a composite key (`audioId` + execution timestamp) would prevent this.

## Test coverage gaps

Not tested: the ordering of states in the chain (WriteInitialRecord before Polly before MarkCompleted), the specific error transition (Catch routes to MarkFailed rather than some other state), or the content of the DynamoDB parameters (item schema, key values, update expressions). A snapshot test or a more targeted definition-string assertion would catch regressions in the state machine topology.
