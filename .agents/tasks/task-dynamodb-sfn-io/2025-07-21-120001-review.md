# DynamoDB metadata tracking for the sleep audio pipeline (v2)

This change adds a DynamoDB table and wires it into the Step Functions state machine as bookend tasks around the existing Polly call: PutItem on entry (with a conditional write preventing in-flight overwrites), UpdateItem on success or failure. Both terminal update states now carry retry with exponential backoff and use `$$.State.EnteredTime` for accurate timestamps. Five CDK assertion tests cover the new resources, permissions, and error handling keywords.

**Watch for:** WriteInitialRecord has no Catch or Retry, so a `ConditionalCheckFailedException` (the "already in-flight" guard firing correctly) fails the entire execution with no graceful signal to the caller — EventBridge will retry the event fruitlessly (confirmed). The `Retry` on MarkCompleted and MarkFailed uses `States.ALL`, which will also retry non-transient errors like validation failures, burning retry budget unnecessarily on errors that won't resolve (likely).

**Verdict**: APPROVED

## High-level view

When the conditional PutItem in WriteInitialRecord rejects a duplicate (record exists and is still PROCESSING), the execution hard-fails with a DynamoDB error. Production deployments will want to distinguish "legitimate duplicate" from "DynamoDB transient error" so the EventBridge retry policy can make an informed decision, but fail-fast is defensible for a dev skeleton.

The retry configuration on both terminal states catches all error classes including non-retryable ones. Since the UpdateItem calls have no ConditionExpression and are idempotent, `States.ALL` won't cause data corruption — but it will retry programming errors (malformed expressions) that can never succeed.

Tests verify resource presence, definition keywords, and IAM policy statements but do not assert the state machine topology (ordering, catch target identity). A structural regression — e.g., accidentally routing Catch to MarkCompleted instead of MarkFailed — would not be caught by the current suite.

## WriteInitialRecord failure mode on conditional rejection

When the ConditionExpression `attribute_not_exists(audioId) OR #s IN (:completed, :failed)` fires, DynamoDB returns `ConditionalCheckFailedException`. Since WriteInitialRecord has neither Retry nor Catch, this propagates as an execution failure:

```
WriteInitialRecord -> ConditionalCheckFailedException -> Execution FAILED
```

If EventBridge retries the event (which it does by default on execution failures), the same conditional rejection will repeat until the retry policy exhausts — wasting EventBridge retry attempts on a condition that will never resolve without external intervention.

A targeted Catch for `DynamoDB.ConditionalCheckFailedException` routing to a terminal Pass state would make the execution end cleanly when a legitimate duplicate is detected, distinguishing it from actual DynamoDB errors.

## Retry scope on terminal states

Both MarkCompleted and MarkFailed retry on `States.ALL`:

```go
markCompleted.AddRetry(&awsstepfunctions.RetryProps{
    Errors:      &[]*string{awsstepfunctions.Errors_ALL()},
    Interval:    awscdk.Duration_Seconds(jsii.Number(2)),
    MaxAttempts: jsii.Number(3),
    BackoffRate: jsii.Number(2.0),
})
```

The updates are idempotent (no ConditionExpression), so retrying non-transient errors won't corrupt data — it just delays the inevitable failure. For production, narrowing to `DynamoDB.InternalServerError` and `DynamoDB.ServiceUnavailableException` would avoid burning retry budget on programming errors that can never succeed.

## Test coverage gap: state machine topology

Not tested: the ordering of states in the chain, the identity of the Catch target (MarkFailed vs any other state), the presence of Retry blocks, or the ConditionExpression content. A snapshot test or a JSON-path assertion on the parsed definition would catch topology regressions without being brittle to formatting changes.
