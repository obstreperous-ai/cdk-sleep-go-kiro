# S3 buckets and EventBridge wiring for the sleep audio pipeline

This change lays the foundation for the event-driven sleep audio pipeline: two S3 buckets (input and output) with encryption, versioning, and public access blocks, an EventBridge rule that matches `Object Created` events from the input bucket, and a placeholder Lambda as the rule target. The implementation follows a TDD approach — seven CDK assertion tests verify the synthesized CloudFormation template. ARCHITECTURE.md gets a new "Implemented Components" section with a current-state mermaid diagram.

**Watch for:** The EventBridge rule's event pattern uses a CDK token for the bucket name filter, which means the test for the rule (`TestEventBridgeRuleExists`) doesn't actually verify the bucket name filter is present — it only checks `source` and `detail-type` (confirmed). The placeholder Lambda uses Node.js 18.x which reaches EOL in the near future; not blocking but worth noting (confirmed). `RemovalPolicy_DESTROY` on both buckets is appropriate for dev but there's no environment gate preventing this from reaching production (likely).

## High-level view

The stack introduces four resources in a linear event chain: an input bucket emits events to EventBridge, a rule matches those events, and a stub Lambda absorbs them. The output bucket sits unconnected for now, a parking space for the future packaging step.

The EventBridge rule filters on bucket name using a CDK token reference (`inputBucket.BucketName()`), which resolves to a CloudFormation `Ref` at synth time. This is correct for deployment but makes it impossible to assert the full event pattern in tests without matching against a token string, which is why the test only checks source and detail-type.

The test suite covers resource existence, encryption configuration, versioning, public access blocks, EventBridge rule shape, target presence, and Lambda properties. It does not cover the relationship between the rule's bucket name filter and the actual input bucket, nor does it verify that *both* buckets (rather than just one) satisfy each property assertion.

The `RemovalPolicy_DESTROY` on both buckets is unconditional. The ARCHITECTURE.md describes multi-environment deployment via CDK context, and nothing in the code currently gates the removal policy on environment.

## EventBridge rule and the bucket name token

The rule's event pattern filters on `detail.bucket.name` using `inputBucket.BucketName()`, which at synth time resolves to a CloudFormation intrinsic (`{"Ref": "SleepAudioInputBucket..."}`) rather than a literal string. This creates a testing blind spot.

`TestEventBridgeRuleExists` asserts:

```go
template.HasResourceProperties(jsii.String("AWS::Events::Rule"), map[string]interface{}{
    "EventPattern": map[string]interface{}{
        "source":      []interface{}{"aws.s3"},
        "detail-type": []interface{}{"Object Created"},
    },
})
```

This passes as long as *any* rule has those two fields, regardless of whether the detail filter is present. If someone accidentally removes the `Detail` block from the rule definition, this test still passes. A stronger assertion would check that the `detail` key exists in the event pattern, even without asserting its exact value:

```go
"detail": map[string]interface{}{
    "bucket": map[string]interface{}{
        "name": assertions.Match_AnyValue(),
    },
},
```

## Test coverage asymmetry on bucket properties

`TestBucketsHaveBlockPublicAccess` passes if at least one bucket has the public access block. If the output bucket lost its `BlockPublicAccess` setting, this test would still pass because the input bucket matches.

The same pattern applies to `TestInputBucketExists` — it asserts encryption and versioning on `AWS::S3::Bucket` using `HasResourceProperties`, which matches against the first bucket that satisfies the predicate. Since both buckets have identical encryption and versioning config, this doesn't uniquely identify the input bucket. The count assertion in `TestOutputBucketExists` (count = 2) partially compensates but doesn't prove each bucket individually carries every required property.

## Encryption: S3-managed vs architecture target

ARCHITECTURE.md's Security Design section states both buckets use "SSE-KMS with per-environment KMS Customer Managed Keys." The implementation uses `BucketEncryption_S3_MANAGED` (AES256). The "Implemented Components" table correctly describes the current state, so the docs are internally consistent — but the gap between target architecture and current implementation should be tracked as a future work item.
