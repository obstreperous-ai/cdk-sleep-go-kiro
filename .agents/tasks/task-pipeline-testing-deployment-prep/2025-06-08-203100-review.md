# Multi-environment support and CDK Pipelines skeleton

This change adds multi-environment stack naming driven by CDK context (`-c env=prod`), a CDK Pipelines CI/CD skeleton in `pipeline.go`, and a significant expansion of test coverage including end-to-end state machine path verification, snapshot testing, and pipeline construct assertions. A deprecation fix for DynamoDB `PointInTimeRecovery` is bundled in. The `cdk.json` app command changes from `go run cdk-base.go` to `go run .` to support the multi-file main package.

Watch for: The pipeline uses a placeholder CodeStar Connection ARN that would fail on deploy (confirmed). The `getEnvContext` helper accepts arbitrary strings without validation, meaning a typo like `-c env=prodd` silently creates an unexpected stack (confirmed). The snapshot test uses byte-level string comparison rather than semantic JSON comparison, making it fragile to formatting changes across CDK library versions (confirmed).

## High-level view

The multi-environment mechanism reads CDK context and defaults to `"dev"`, renaming the default stack from `CdkBaseStack` to `SleepAudioPipeline-dev` — a breaking change for any existing deployment targeting the old stack name. No validation constrains the env value.

The CDK Pipelines skeleton is deliberately incomplete — a placeholder connection ARN, no approval gates, no cross-account setup. It's wired into `main()` behind a `pipeline=true` context flag so it doesn't interfere with normal synthesis. The synth step is missing `go mod download` and `GOPROXY=direct`, which would cause failures in a fresh CodeBuild environment.

The test suite gains integration-style tests that verify state machine transition chains via regex matching on the serialized `Fn::Join` definition string. This couples tests to the serialization format CDK produces internally — a CDK version upgrade could break these tests without changing behavior.

## getEnvContext and the stack rename

The `getEnvContext` function concatenates the context value directly into the stack ID with no validation. CloudFormation stack names are limited to 128 alphanumeric/hyphen characters, so passing something like `-c env="my stack!!"` would cause a deploy-time error with a confusing message. A simple `regexp.MustCompile("^[a-z]+-?[a-z]+$")` check would catch misuse early.

The stack ID change from `CdkBaseStack` to `SleepAudioPipeline-dev` means any environment that previously deployed via `cdk deploy CdkBaseStack` now has an orphaned stack. The new stack would create duplicate resources rather than updating existing ones.

## Pipeline skeleton security posture

The pipeline uses a hardcoded placeholder ARN:

```go
ConnectionArn: jsii.String("arn:aws:codestar-connections:us-east-1:123456789012:connection/placeholder"),
```

There's no guard preventing accidental deployment. A CDK `Annotations.Of(stack).AddError()` when the ARN contains "placeholder" would make synthesis fail loudly rather than deploying a broken pipeline.

The pipeline's CodeBuild project (created implicitly by CDK Pipelines) gets broad IAM permissions for `cdk deploy`. For the eventual production deployment, the pipeline's execution role effectively has admin-like access to the target account — worth constraining via a permissions boundary when moving past skeleton stage.

## Synth step completeness

```go
Commands: &[]*string{
    jsii.String("go test ./..."),
    jsii.String("npx cdk synth"),
}
```

The `go test` step runs before `npx cdk synth` triggers the app command (which includes `go mod download`). In a fresh CodeBuild environment without cached modules, `go test` would need to implicitly download dependencies. The step also doesn't set `GOPROXY=direct`, which the project's context.json documents as required. Adding `go mod download` and the env var to the ShellStep (or using `InstallCommands`) would make this reliable.

## Snapshot test design

The byte-level string comparison:

```go
if string(snapshotBytes) != string(goldenBytes) {
    t.Fatalf("snapshot mismatch...")
}
```

breaks if a CDK library upgrade reorders keys in the JSON output, even though the template is semantically identical. Unmarshaling both into `map[string]interface{}` and using `reflect.DeepEqual` would be more robust.

The auto-create-on-first-run behavior (`t.Skip` when the file doesn't exist) means CI silently passes if someone accidentally deletes the snapshot file: the first CI run recreates it and skips comparison. Consider failing instead of skipping when the snapshot is missing in CI (e.g., check a `CI` env var).

## State machine regex tests

The regex patterns (e.g., `WriteInitialRecord.*"Next".*ValidateInput`) depend on the concatenation boundaries CDK chooses for the `Fn::Join` array. A CDK version upgrade could split the definition string at a different boundary, causing two states to land in different array elements — breaking the `Match_ArrayWith` assertion even though the state machine is functionally identical. There's no first-class Step Functions definition introspection API in the CDK assertions library, so this is an accepted trade-off, but these tests are more CDK-version-coupled than typical `HasResourceProperties` checks.

## Pipeline test coverage gaps

The three pipeline tests verify synthesis and stage structure but don't test:

- That the deploy stage actually contains the application stack
- That the pipeline name includes the environment suffix
- That passing different `EnvName` values produces different pipeline names

These would be straightforward additions using `HasResourceProperties` with a `PipelineName` matcher.
