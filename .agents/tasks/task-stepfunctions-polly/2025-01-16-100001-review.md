# Step Functions state machine with Polly task (v2)

The placeholder Lambda is replaced by a Step Functions state machine with a single `CallAwsService` task invoking Polly's `synthesizeSpeech` API. EventBridge targets the state machine; CloudWatch Logs (ALL, with execution data) and X-Ray tracing are wired in. Test suite now asserts specific Polly integration details rather than just resource existence.

Watch for: The Polly task still uses hardcoded parameters and discards its output (confirmed) — acknowledged as intentional for the skeleton per issue requirements. No remaining blocking concerns from this pass.

**Verdict**: APPROVED

## High-level view

All four v1 issues have been addressed. The definition test now verifies `Fn::Join` array elements contain `polly:synthesizeSpeech` via `Match_StringLikeRegexp`, catching regressions where the Polly action changes or disappears. The IAM test asserts `Action: "polly:synthesizeSpeech"` scoped to roles referencing the state machine logical ID — it will no longer pass if the action is accidentally removed. The Standard-vs-Express decision is documented with a code comment explaining the rationale and when to switch.

The remaining known limitation — hardcoded Polly parameters ignoring execution input — is explicitly scoped out as appropriate for the skeleton iteration and documented in the feature findings.
