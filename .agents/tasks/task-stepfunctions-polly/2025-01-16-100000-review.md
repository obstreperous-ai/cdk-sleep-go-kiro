# Step Functions state machine with Polly task replacing placeholder Lambda

The placeholder Lambda is removed and replaced with a Step Functions state machine containing a single `CallAwsService` task that invokes Polly's `synthesizeSpeech` API. The EventBridge rule now targets the state machine instead of the Lambda. CloudWatch Logs (ALL level with execution data) and X-Ray tracing are wired in. Six new CDK assertion tests cover the state machine's existence, definition, logging, IAM, EventBridge wiring, and Lambda removal.

Watch for: The Polly task uses hardcoded text/voice parameters rather than consuming the EventBridge event payload (confirmed), which means the state machine ignores its input entirely. The IAM policy grants `polly:synthesizeSpeech` plus logs and X-Ray permissions to `Resource: "*"` in a single merged statement (confirmed) — acceptable for a skeleton but worth noting for future tightening. The `TestStateMachineIAMRole` test doesn't assert the specific Polly action, only `Effect: Allow` + `Resource: *` (confirmed), so it would pass even if the action changed.

## High-level view

The state machine is a single-task skeleton: EventBridge triggers it, it calls Polly with static parameters, and the execution ends. There's no consumption of the S3 event payload that triggered it — the `Text`, `OutputFormat`, and `VoiceId` are literals baked into the definition string. Any upload produces the same Polly call regardless of content.

CDK merges logs, X-Ray, and Polly permissions into one `Resource: "*"` statement. The EventBridge-to-Step-Functions wiring uses a separate Events role scoped to `states:StartExecution` on exactly the state machine ARN. As more tasks are added, the `"*"` resource scope on the execution role should be tightened per-task.

The test suite stays at the "resource exists with these properties" level. `TestStateMachineDefinitionContainsPollyTask` only asserts that `DefinitionString` is present, not that it contains a Polly action. `TestStateMachineIAMRole` asserts shape but not the specific action string — it would pass even if the Polly permission were removed.

## Polly task with static parameters

The Polly `synthesizeSpeech` call is defined with literal values:

```go
Parameters: &map[string]interface{}{
    "Text":         "Welcome to your sleep audio session. Relax and breathe deeply.",
    "OutputFormat": "mp3",
    "VoiceId":      "Joanna",
},
```

The EventBridge event payload (which contains the S3 bucket name, object key, and event metadata) arrives as execution input but is never referenced. The next iteration needs to wire `$.detail.object.key` or similar into the parameters for the pipeline to actually process uploaded files.

The Polly response (an audio stream) is also not stored anywhere — the task ends successfully but the audio bytes are dropped. The state machine currently proves the SDK integration works end-to-end but produces no durable output.

## Standard vs. Express state machine type

The ARCHITECTURE.md design section references "Express Workflows" multiple times as the intended execution model for short-duration, high-throughput audio processing. The CDK code does not set `StateMachineType` on the `StateMachineProps`, which means CDK defaults to `STANDARD`. Standard state machines have different pricing (per state transition), a 365-day execution history limit, and exactly-once semantics — versus Express which is priced per duration, has a 5-minute max duration, and at-least-once semantics.

If Express is the target (as documented), the prop should be:
```go
StateMachineType: awsstepfunctions.StateMachineType_EXPRESS,
```

Express workflows use CloudWatch Logs as their *only* history mechanism (no execution history API), making the log group configuration even more critical. The logging is already in place, so switching to Express would be a one-line change.

## Test coverage gaps

`TestStateMachineDefinitionContainsPollyTask` asserts `DefinitionString: Match_AnyValue()`, which would pass for any state machine with any definition. A stronger assertion would use `Match_StringLikeRegexp` on the DefinitionString to confirm the presence of `polly` or `synthesizeSpeech`.

`TestStateMachineIAMRole` matches `Effect: Allow` and `Resource: "*"` but doesn't include the action. This means the test wouldn't catch a regression where the Polly permission was accidentally removed, since the logs/X-Ray permissions also carry `Effect: Allow` + `Resource: "*"`.
