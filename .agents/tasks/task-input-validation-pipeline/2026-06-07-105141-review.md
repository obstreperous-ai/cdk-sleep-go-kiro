# Input validation via Step Functions Choice state and Lambda runtime wiring

The change adds a two-layer input validation strategy to the audio processing pipeline. A Step Functions Choice state (`ValidateInput`) checks file extensions at the orchestration layer before invoking any compute, and the Lambda handler itself validates required fields and extensions as a defense-in-depth measure. The Lambda is also wired to the actual Lambda runtime API (`lambda.Start`), replacing the previous skeleton. An `OUTPUT_BUCKET_NAME` env var is plumbed through to the Lambda. Five new CDK assertion tests cover the validation states.

**Watch for:** The `audioId` used as the DynamoDB key is the raw S3 object key (the full path including prefix), not a derived identifier — this means the Choice state's `StringMatches` on `*.mp3` validates the **entire** key path (not just the filename), which could produce false negatives for keys containing dots in directory segments (confirmed). The Lambda lacks any grant to the output bucket despite receiving `OUTPUT_BUCKET_NAME` (confirmed). Case sensitivity between the two validation layers is inconsistent (likely).

## High-level view

The validation gate is a Choice state inserted between `WriteInitialRecord` and `ProcessAudio`. It checks `$.detail.object.key` against five glob patterns (`*.mp3`, `*.wav`, etc.) and routes non-matching keys directly to `MarkFailed`. No compute is invoked for rejected files, and `Otherwise` serves as the default-deny branch.

The Lambda handler validates `audioId`, `bucket`, and `objectKey` are non-empty, then checks the file extension (case-insensitive via `strings.ToLower`). This creates a mismatch with the Choice state, which uses `StringMatches` — a case-sensitive comparator in Step Functions. A file named `recording.MP3` would pass the Lambda but fail the Choice state.

The `OUTPUT_BUCKET_NAME` environment variable is added but never used in the handler code and no IAM grant exists for the Lambda to write to that bucket. The grant gap means the next iteration that tries to write will fail at runtime with `AccessDenied`.

The CDK tests verify structural properties of the synthesized template but no unit test file exists for the Lambda handler's branching validation logic.

<summary>Issues (5)</summary>

1. **Case-sensitive mismatch between validation layers** — Step Functions `StringMatches` is case-sensitive; the Lambda normalizes to lowercase. Files with uppercase extensions (`.MP3`, `.Wav`) will be rejected by the Choice state even though the Lambda would accept them. Either add uppercase variants to the Choice conditions or apply an intrinsic function to lowercase the key before matching. (likely)
2. **Glob matches full object key, not just extension** — `*.mp3` matches any key ending in `.mp3`, but a key like `mixes.mp3.backup/final` would also pass despite not being an mp3 file. Consider anchoring the pattern or noting that S3 key conventions prevent this in practice. (possible)
3. **No IAM grant for Lambda to write to output bucket** — `OUTPUT_BUCKET_NAME` is provided but `outputBucket.GrantWrite(processorLambda)` (or similar) is missing. The Lambda will get `AccessDenied` when it attempts to use this bucket. (confirmed)
4. **No unit tests for Lambda handler validation logic** — The Go handler has branching validation logic (required fields, extension check) with no corresponding `_test.go` file. CDK assertions verify infrastructure, not runtime behavior. (confirmed)
5. **`audioId` is the raw S3 object key** — The state machine uses `$.detail.object.key` as both `audioId` and `objectKey` in DynamoDB and the Lambda payload. This conflates identity with location. If the same logical audio is re-uploaded to a different key, it creates a separate record rather than updating the existing one. (confirmed — may be intentional)

## Validation gate: Choice state design and glob semantics

The Choice state uses `StringMatches` with patterns like `*.mp3`. The `*` wildcard matches any sequence of characters, so `uploads/user123/recording.mp3` matches correctly. The case sensitivity is the primary concern: `StringMatches` is case-sensitive per the Step Functions spec, so `something.MP3` fails the check.

`WriteInitialRecord` executes *before* `ValidateInput`. This means a DynamoDB record with `status=PROCESSING` is created even for files that immediately fail validation. The table will accumulate `FAILED` records for every non-audio file uploaded to the bucket. The EventBridge rule doesn't filter by key prefix or extension — it fires on all `Object Created` events. If the bucket receives non-audio uploads frequently, this produces metadata noise.

## Lambda validation and the output bucket gap

The extension check normalizes to lowercase via `strings.ToLower(filepath.Ext(...))`, which is more permissive than the Choice state's case-sensitive glob. Since the Choice state runs first, the Lambda's case-insensitivity is unreachable for uppercase extensions — they never arrive at the Lambda.

`OUTPUT_BUCKET_NAME` exists in the Lambda's environment but is never read by the handler. More importantly, `outputBucket.GrantWrite(processorLambda)` is absent from the CDK stack:

```go
// cdk-base.go — grants present:
metadataTable.GrantReadWriteData(processorLambda)
// missing: outputBucket.GrantWrite(processorLambda)
```

When future code adds S3 writes, the IAM policy won't allow it without adding this grant.

## State machine wiring and error propagation

```
                       ┌──────── [catch] ──────────┐
                       │                           │
WriteInitialRecord → ValidateInput                 │
                       │ valid        │ otherwise   │
                       ▼              ▼             │
                  ProcessAudio → MarkFailed ← ─────┘
                       │              │
                       ▼              ▼
                    PollyTask    NotifyFailed
                       │ catch──→MarkFailed
                       ▼
                  MarkCompleted
                       ▼
                  NotifyCompleted
```

`ProcessAudio` and `PollyTask` both catch to `MarkFailed`. The `ValidateInput` Choice state's `Otherwise` also routes to `MarkFailed`. `MarkFailed` retries (3 attempts, exponential backoff) for transient DynamoDB throttling. `NotifyFailed` similarly retries and catches to a Pass fallback, preventing notification failure from failing the entire execution.

`WriteInitialRecord` has no `AddCatch`. If the DynamoDB `PutItem` fails (the `ConditionExpression` throws `ConditionalCheckFailedException` for duplicate in-flight uploads), the state machine execution fails with an unhandled error — no status update, no notification. This is a pre-existing pattern from a prior PR, so out of scope here, but the validation change doesn't address it.

## Test coverage gaps

**Not tested:** The Lambda handler's Go logic (field validation, extension normalization, error return format) has no unit tests. There is no `lambda/processor/main_test.go`. The handler contains branching logic that determines whether the pipeline succeeds or fails, and CDK assertions cannot verify that the Lambda returns the right error format for the state machine's Catch to interpret.
